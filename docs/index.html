Comment ça marche — en 1 minute

Chaque 1er du mois, l'outil lit deux signaux sur le prix BTC :

  • Signal tendance : BTC a-t-il monté sur les 11 derniers mois ? → ACHAT ou CASH.
  • Signal valorisation : BTC est-il trop cher ou à bon prix par rapport à sa trajectoire long-terme ? → ACHAT ou CASH.

Les 2 signaux combinés te donnent une seule consigne : garder 100 %, 50 % ou 0 % de ton capital en BTC. Voici la règle :

Signal tendance Signal valorisation Ta position BTC
ACHAT ACHAT
🟢 100 % BTC
climat haussier
CASH ACHAT
🟡 50 % BTC
signaux opposés (prudence partielle)
ACHAT CASH
🔴 0 % BTC

baisse confirmée

C'est tout. Pas d'autre règle, pas d'exception, pas de « oui mais ». La consigne change en moyenne ~1 fois par an. Les autres mois, tu n'as rien à faire.

Questions fréquentes

Méthode ouverte, gratuite, communautaire. L'idée, c'est de gagner ensemble.

C'est gratuit ? Je paye quoi ?

Oui, entièrement gratuit. ChillBTC est open-source sous licence CC BY-NC-SA 4.0. Le code, les données et les paramètres sont publics sur GitHub. Pas d'abonnement, pas de compte premium, pas de token à acheter. Seule contrainte de la licence : usage non-commercial uniquement — tu ne peux pas revendre la méthode ou en faire un service payant.

Qui décide du signal chaque mois ? Quelqu'un peut-il le truquer ?

Personne n'a la main sur le signal affiché. Le code qui calcule le dosage est public sur GitHub, le workflow qui publie la page chaque 1er du mois est auditable ligne par ligne, et les paramètres sont figés jusqu'à la prochaine revue annuelle. Tu peux toi-même cloner le dépôt et vérifier que le signal que tu vois correspond exactement à ce que le code produit.

Je peux proposer une nouvelle règle ou une amélioration ?

Oui, et c'est l'intérêt d'une méthode ouverte. Deux garde-fous stricts encadrent les propositions :

  • Rester sur la cadence « un coup d'œil par mois » : pas de décision intra-mois, pas de stop-loss, pas de règle qui demande de surveiller le prix au quotidien.
  • Pas de sur-optimisation : la règle doit marcher sur une plage large de paramètres, pas sur un pic isolé de la simulation historique. Le template d'issue feature_request sur GitHub te guide sur les checks à valider.
Pourquoi open-source plutôt que payant ?

Parce que la force d'une méthode, c'est la communauté qui l'examine. Une stratégie secrète, calibrée seule dans son coin, finit par dériver : l'auteur se convainc qu'elle marche et ne voit plus ses biais. Ouvrir le code, c'est accepter la critique, détecter les bugs, mélanger les idées de tout le monde. L'idée, c'est de gagner ensemble.

Je suis le bienvenu même si je ne suis pas développeur ?

Oui, évidemment. Tu peux contribuer sans écrire une ligne de code :

  • Signaler un bug ou une ambiguïté dans la documentation — les templates d'issue GitHub te guident pas à pas.
  • Partager ton retour d'expérience quand tu appliques la méthode au fil du temps.
  • Traduire la doc si tu maîtrises une autre langue.
  • Relire la méthodologie et pointer ce qui reste opaque pour un non-expert.
Comment la méthode évolue-t-elle dans le temps ?

Elle est figée jusqu'à la prochaine revue annuelle (1er janvier). Entre deux revues, aucun paramètre ne bouge — c'est le garde-fou contre les ajustements au feeling. À la revue annuelle, la communauté peut proposer des ajustements via GitHub (issues et pull requests). Seul un événement exceptionnel (faillite de plateforme, dépegging de stablecoin, changement réglementaire majeur) peut déclencher une revue hors cycle, documentée publiquement dans le dépôt.
